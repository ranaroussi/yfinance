## Purpose

This file tracks whether the large refactor in this fork appears to resolve any currently open upstream issues. These statuses started as an audit snapshot and are now being upgraded with issue-specific verification tests under `tests/issues`. A status of `resolved in fork` means the current code now has either direct regression coverage or a targeted verification test that reproduces the reported failure mode and passes on this fork.

## Status Legend

- `resolved in fork`: strong code and/or test evidence that the refactor addresses the issue
- `needs reproduction`: the refactor touched the relevant area, but the exact issue was not verified end-to-end
- `not resolved`: the issue still appears open in this fork, or the fork explicitly chose a different behavior
- `not addressed`: no meaningful evidence that the refactor targeted this issue
- `invalid usage`: the report is based on an incorrect ticker, unsupported input, or user error rather than a library defect
- `new feature requests`: feature request, question, or docs request outside this bug-fix audit

## Fork Areas That Changed

- history/download stack split from one large module into dedicated fetch, repair, reconstruct, dividend, split, and capital-gains modules
- HTTP/session stack moved to curl_cffi with centralized cookie, crumb, proxy, and retry handling
- timezone and date-boundary handling were refactored across history, fast_info, and utilities
- quote/info, funds, options, search, live, and screener modules were reorganized and partially hardened
- test coverage expanded significantly, especially for price repair, cache, ticker core behavior, and intraday/dividend edge cases
